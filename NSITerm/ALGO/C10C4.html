    Introduction⚓︎

    Histoire

    La programmation dynamique est un concept algorithmique introduit au début des années 1950 par le mathématicien américain Richard Bellman.

    Comme la méthode « diviser pour régner», elle consiste à décomposer un problème difficile en sous-problèmes plus simples à résoudre et à les utiliser pour trouver la meilleure solution au problème initial.

    La principale différence entre ces deux approches est l'usage de la mémoïsation, c'est-à-dire l'enregistrement des valeurs intermédiaires des sous-problèmes dans un cache pour éviter qu'ils ne soient calculés plusieurs fois.

    Retour sur la suite de Fibonacci⚓︎

    Au chapitre I on a rencontré cette suite:

    la suite de Fibonacci définie par :

    • \(F_0 = 0\)
    • \(F_1 = 1\)
    • \(\forall n \in \mathbb{N}, F_{n+2} = F_{n+1}+F_n\)

    On avait alors choisit de résoudre ce problème avec une approche récursive naïve ayant des performances très mauvaises. La faute aux nombreux calculs redondants.

    Programmation dynamique⚓︎

    Approche Top-Down⚓︎

    La première approche que nous utiliserons est l'approche Top-down. On part des sous problèmes les plus grands vers les plus petits.
    L'idée est similaire à l'approche naïve à une différence fondamentale près: on sauve les solutions des sous problèmes rencontrés. Cela nous permet de calculer chaque sous problème exactement une fois.

    Algorithme⚓︎

    Le choix est fait ici d'initialiser le cache servant à accueillir les valeurs de la mémoïsation dans fibo puis d'appeler TopDown_fibo.

    Un exemple en détail: le rendu de monnaie⚓︎

    On l'a vu en activité la stratégie gloutonne, bien qu'intéressante, ne donne pas toujours une solution optimale, notamment si le système de pièces n'est canonique.

    Essayons un autre algorithme, récursif, qui donnerait la solution optimale quelque soit le système monétaire utilisé. Le cas de base est trivial.

    • Si la somme à rendre est nulle, aucune pièce n'est nécessaire.
    • Sinon, on considère chaque pièce \(p\) et on calcule récursivement le nombre de pièces minimal pour rendre \((somme-p)\)

    Comme exemple, supposons le système formé de pieces = [1, 4, 5] et la somme à atteindre: 13.

    Formalisation

    L'idée est la suivante : pour rendre la somme \(n\) de façon optimale il faut rendre une pièce \(p\) et la somme \(n-p\) de façon optimale. Pour cela, il faut tester toutes les pièces \(p\) possibles et prendre celle qui minimise le nombre de pièces pour rendre pour la somme \(n-p\).

    Une solution récursive pourrait être:

    • rendu(pieces, 0) = 0
    • rendu(pieces, somme) = min(1 + rendu(pieces, somme - p)) avec \(p\in pieces\) lorsque cela est possible.

    L'implémentation pourrait être:

    Problème

    Les performances de ce code sont épouvantables. Le problème vient des nombreux appels récursifs avec les mêmes paramètres. Les sous problèmes se chevauchent.

    Essayons la programmation dynamique, où une idée forte est la mémoïsation. On calcule une fois chaque sous problème et on sauvegarde le résultat dans un tableau.

    def rendu(pieces, s):
    """Version récursive avec mémoïsation"""
    memo = [None] * (s + 1) # intialisation du tableau avec des None 
    return rendu_topdown(pieces, s, memo)

def rendu_topdown(pieces, s, memo):
    if memo[s] is not None: # si on a déjà calculé le nombre optimal de pièces pour rendre la somme s
        return memo[s]      # on le renvoie directement
    elif s == 0:
        memo[s] = 0
        return 0
    else:
        nb_pieces = s     # nb_pieces = 1 + 1 + ... + 1 dans le pire des cas
        for p in pieces:
            if p <= s:   # inutile de tester une pièce dont la valeur dépasse la somme s à rendre
                nb_pieces = min(nb_pieces, 1 + rendu_topdown(pieces, s-p, memo))
        memo[s] = nb_pieces
        return nb_pieces

    Commentaires

    Le système est composé de pièces \(\left[1,\cdots p_i, \cdots ,p_{max}\right]\).

    On part de l'hypothèse où on sait rendre de façon optimale toutes les valeurs strictement inférieures à somme. Pour rendre somme, il faut au moins une pièce, à prendre parmi les pièces disponibles du système monétaire. Une fois cette pièce choisie, la somme restante est inférieure (strictement) à somme, donc on sait la rendre de façon optimale. Il suffit d'essayer toutes possibilités parmi les pièces du système proposé:

    \(nb\_ piece[somme]=1+min(nb\_piece[somme-p_i]\) pour \(p_i\in \left[1\cdots p_{max}\right]\)

    def rendudynamique1(pieces, somme):
    """
    Renvoie le nombre de pieces minimal 'nb_piece' nécessaire pour former 'somme'
    """
    nb_piece = [0] * (somme+1) # initialisation du cache
    solution = [[]] * (somme+1) # liste pour la solution
    for n in range(1, somme+1): # on effectue le calcul de tous les ss pb de 1 jusqu'à somme
        nb_piece[n] = n
        solution[n] = [1] * n
        for p in pieces:
            if p <= n and nb_piece[n] > 1+nb_piece[n-p]:
                nb_piece[n] = 1+nb_piece[n-p]
                solution[n] = [c for c in solution[n-p]] + [p]
    return solution[somme]
